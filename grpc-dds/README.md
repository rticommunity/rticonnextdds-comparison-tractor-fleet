# Hybrid — DDS Pub-Sub + gRPC Request-Reply

A robot fleet demo that uses **DDS (Connext)** for all publish-subscribe
data flows and **gRPC** for request-reply interactions (operator commands
and charging station negotiations).

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        Data Plane (DDS)                         │
│  KinematicState · OperationalState · Intent · Telemetry         │
│  VideoFrame (partition-based) · StationStatus                   │
│  Discovery: DDS SPDP (automatic multicast)                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    Control Plane (gRPC)                          │
│  RobotCommand.SendCommand (UI → robot, unary)                   │
│  ChargingStation.{RequestSlot,ConfirmSlot,ReleaseSlot}          │
│  Discovery: Zeroconf / mDNS                                    │
└─────────────────────────────────────────────────────────────────┘
```

### Why this split?

| Pattern | DDS | gRPC |
|---------|-----|------|
| 1-to-N pub-sub | Native multicast, QoS profiles, auto-discovery | Must emulate with per-peer streaming RPCs |
| Request-reply | Works (rti.rpc) but adds complexity | Native, simple, well-tooled |
| Service discovery | Built-in SPDP | Needs Zeroconf / consul / etc. |
| Video streaming | Partition-based writer matching | Server-streaming RPC |

This approach uses each technology where it excels:
- **DDS** for high-frequency telemetry and state (10 Hz kinematic, 1 Hz telemetry)
- **gRPC** for low-frequency commands and transactional operations (charging slots)

## Components

| File | Role |
|------|------|
| `robot_node.py` | Tractor robot: DDS publishers (6 topics incl. CoveragePoint) + DDS readers (4 topics) + gRPC command server + gRPC charging client |
| `charging_station.py` | Station: gRPC server for slot negotiation + DDS StationStatus publisher |
| `robot_ui.py` | Dashboard: DDS readers (7 topics incl. CoveragePoint) + gRPC command client (Zeroconf-discovered) + coverage map overlay |
| `fleet_discovery.py` | Zeroconf mDNS helper for gRPC endpoint discovery |
| `robot.proto` | gRPC service & message definitions (commands + charging) |
| `robot_types.xml` | DDS type definitions (XML Application Creation) |
| `robot_qos.xml` | DDS QoS profiles (incl. CoverageProfile with PERSISTENT durability) |
| `robot_types.py` | Python DDS types (generated by rtiddsgen) |
| `fleet_config.sh` | Transport config for this approach (DDS domain + gRPC base ports); sources `../shared/fleet_common.sh` |
| `run_demo.sh` | Uniform launch script: `all`, `robots`, `stations`, `ui`, `robot <name>`, `station <name>` |
| `run_persistence_service.sh` | Launch RTI Persistence Service for durable CoveragePoint data |
| `stop_demo.sh` | Kill all running processes and free ports |

**Shared assets** (in `../shared/`):

| File | Purpose |
|------|---------|
| `fleet_common.sh` | Single source of truth for robot names, station positions, UI port — shared by all approaches |
| `video_renderer.py` | PyBullet headless 3D renderer — compound tractor bodies, waypoint beacons, text overlay |
| `arena_ground.urdf` | URDF arena ground plane for PyBullet |
| `field1_map.jpg` / `field1_texture.jpg` | Ground textures for canvas map and 3D renderer |

## Quick Start

```bash
# 1. Generate protobuf code
./run_protoc.sh

# 2. Run everything (stations + robots + UI)
#    DDS type support is generated automatically on first run
./run_demo.sh all
```

Individual components:

```bash
./run_demo.sh stations     # all charging stations
./run_demo.sh robots       # all robots
./run_demo.sh ui           # dashboard UI
./run_demo.sh robot tractor1   # single robot (port auto-derived from name)
./run_demo.sh station station1 # single station (dock coords auto-derived)
```

The CLI is **uniform** across all three approaches — same commands, same
arguments.  Robot/station names and dock coordinates come from
`../shared/fleet_common.sh`.

Then visit http://localhost:5000

## Comparison with Other Approaches

| Metric | gRPC (`grpc/`) | Hybrid (this) | DDS (`dds/`) |
|--------|--------------------|----------------------|-------------------|
| Pub-sub transport | gRPC streaming (emulated) | DDS (native) | DDS (native) |
| Command transport | gRPC unary | gRPC unary | rti.rpc Requester |
| Discovery | Zeroconf only | DDS SPDP + Zeroconf | DDS SPDP only |
| Connections (5 robots) | ~25 gRPC channels, 105 streams | 7 DDS readers + 5 gRPC | 7 DDS readers + 1 Requester |
| Threads (5 robots) | ~22 | ~8 | ~4 |
| Video delivery | gRPC streaming | DDS partition matching | DDS partition matching |
| Coverage tracking | gRPC StreamCoverage | DDS CoveragePoint topic | DDS CoveragePoint topic |
| Extra dependencies | grpc, zeroconf | grpc, zeroconf, rti.connextdds | rti.connextdds |
